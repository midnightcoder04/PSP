# Implementation Plan: Slide Navigation, Sequential Reveal, Exercise UX Rework & Testimonials

**Branch**: `003-slide-nav-ux-rework` | **Date**: 2026-05-15 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-slide-nav-ux-rework/spec.md` + user direction (2026-05-15)

## Summary

Iteration 4 reshapes the participant journey around explicit pacing (one slide per exercise, locked sections), upgrades five exercise interaction patterns (auto-counted ranking, value-budget tracker, drag-and-drop ranking, structured multi-question text, rating picker), and adds an end-of-course testimonials feature surfaced in both Admin and Facilitator dashboards.

Architecturally this is **additive**: one new DB table (`testimonials`), one optional column on `exercises` (`slide_group`), one optional JSONB field on `sections.framing` (`reading_material`), two new exercise types (`structured-text`, `rating-picker`), and one new optional behavioural flag on the existing `ranking` type (`interaction: 'drag' | 'buttons'`). No reshaping of `responses` / `progress` schemas; no new RPCs in the critical path. One new dependency (`@dnd-kit/core` + `@dnd-kit/sortable`, ~8 KB gz).

The plan is broken into **eight workstreams** (WS-1…WS-8) so they can be parallelised by independent contributors:

| WS | Scope | Owner concern |
|----|-------|---------------|
| WS-1 | Slide navigation, intro slide, section locking | Pacing UX |
| WS-2 | WATUSI auto-count + linked slide-group | Attitudes UX |
| WS-3 | Values shopping spree widget + gated proceed | Values UX |
| WS-4 | Drag-and-drop ranking (Roles) | Roles UX |
| WS-5 | Structured multi-question text exercises | Past Exp/Contract/Mission UX |
| WS-6 | Rating picker (1–5) exercise type | Transferable Skills UX |
| WS-7 | Goal Setting pattern application | Goals UX |
| WS-8 | Testimonials (DB + RLS + dashboards + course-end CTA) | Closing the loop |

## Technical Context

**Language/Version**: TypeScript 5.5 + React 18.3 + Vite 5.4 *(unchanged)*
**Primary Dependencies**:
  - existing: `@supabase/supabase-js` v2.105, `react-router-dom` v6.26, `recharts` v2.12
  - **new**: `@dnd-kit/core` v6.x + `@dnd-kit/sortable` v8.x (~8 KB gz combined)
**Storage**: Supabase Postgres 15 — additions: 1 new table (`testimonials`), 1 optional column on `exercises` (`slide_group int`), JSONB field additions on `sections.framing` and on `exercises.content_json` (no schema migration needed for JSONB).
**Testing**: Vitest + `@testing-library/react`, plus `@dnd-kit/core` keyboard-accessibility patterns; integration tests for testimonials RLS via the same hosted-DB pattern as `scripts/rpc.test.ts`.
**Target Platform**: Web — Chrome 110+, Firefox 115+, Safari 16+ (drag-and-drop pointer events supported in all three).
**Project Type**: Single-page web application.
**Performance Goals**: p95 initial load ≤ 2 000 ms (4G); TTI ≤ 3 500 ms; Supabase RPC p99 ≤ 500 ms *(unchanged)*. Slide transitions ≤ 16 ms per frame (60 fps).
**Constraints**:
  - Net JS additions ≤ 15 KB gzipped (NFR-001).
  - DB additions: 1 new table + 1 nullable column + JSONB field extensions. Zero-downtime migration (additive).
  - All new exercise types MUST share the existing `(participant_id, exercise_id, session_id)` response shape.
  - Slide navigation MUST NOT hijack global arrow keys (would break text inputs).
  - Drag-and-drop MUST have a keyboard fallback (Up/Down + Space) for accessibility.
**Scale/Scope**: < 500 total users; 5–30 concurrent participants per session; ≤ 50 testimonials/session expected. Bundle size budget per workstream tracked in research.md R8.

## Phase 0 — Research

See [research.md](research.md). Highlights:

- **R1** — Slide transitions: pure CSS `transform: translateX()`, no carousel library.
- **R2** — `reading_material` shape: optional `{ title; content; url? }` on `framing` JSONB.
- **R3** — Section locking: client-side derivation from already-loaded `progress` rows.
- **R4** — WATUSI auto-count: derive from checklist response by `id` prefix; counts read-only.
- **R5** — Values budget: floating widget via fixed-position component; total persisted with rows.
- **R6** — Drag-and-drop: `@dnd-kit/sortable` (accessible, ~8 KB gz, MIT, maintained).
- **R7** — Structured-text and rating-picker: new exercise types added to `exercises.type` enum; legacy data isolated under `_legacy` key.
- **R8** — Bundle budget per WS.
- **R9** — Testimonials RLS: three policies on `testimonials` — self, facilitator-of-session, admin-all.
- **R10** — Multi-question completion semantics: all questions required in UI; DB allows null sub-answers for autosave.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| **Bundle delta exceeds Constitution §IV 10 KB default** (NFR-001 sets 15 KB; estimated participant-route delta ~14 KB after lazy-loading admin/facilitator TestimonialsPage) | Iteration 4 ships *eight* user-facing capabilities (slide nav, intro slide, section locking, two new exercise types, drag-and-drop ranking, value-budget widget, WATUSI count derivation, testimonials submission flow). Splitting into a sub-iteration to stay under 10 KB would deliver each capability in isolation and require ≥ 2 PSP™ workshop cycles before participants see end-to-end value. Constitution §IV permits this overage when explicitly tracked here. | (a) Drop drag-and-drop → A11Y regression for Roles; rejected on Constitution §III. (b) Defer testimonials → loses end-of-course closure that drove the user direction; rejected. (c) Use plain `<select>` rating picker instead of segmented radio group → worse mobile UX, fails AC-8 intent; rejected. Lazy-loading the admin/facilitator TestimonialsPage chunks (~1 KB) is included in the mitigation; T077 verifies the final delta. |
| New dependency `@dnd-kit/core` + `@dnd-kit/sortable` (~8 KB gz) | AC-6 mandates drag-and-drop ranking with full keyboard accessibility. | Native HTML5 drag-and-drop: lacks touch and keyboard support. `react-beautiful-dnd`: abandoned upstream, no React 18 strict-mode support. Custom impl: re-derives ~600 LOC of well-tested accessibility primitives (focus management, screen-reader announcements). |
| Two new exercise types (`structured-text`, `rating-picker`) | AC-7/AC-8 specifically reshape participant input semantics. A single generic "form" type would conflate concerns. | A `form`-typed exercise with discriminated content unions would push the union complexity into every renderer. Two narrow types with their own renderers keeps each implementation small and individually testable. |
| **CI performance gate deferred to Iteration 5** | Constitution §IV mandates CI-level blocking of performance regressions. Iteration 4 ships T077 (bundle-delta verification) and adds a manual 60-fps spot-check (T029) but does not stand up a Lighthouse-in-CI workflow. | A full Lighthouse-CI rollout requires: GitHub Actions secrets for a preview env, baselines per route, alert routing — none of which are in place. Standing it up inside Iteration 4 would dwarf the user-facing scope. T077 + the existing `npm run build` size check are the interim gate; full CI perf gate scheduled for Iteration 5 (open work item, to be filed in `specs/004-…/` once that branch exists). |
